# ADR-001 至 ADR-010：电商 AI 客服平台架构基线

状态：已接受  
日期：2026-07-12

| 编号 | 决策 | 后果 |
|---|---|---|
| ADR-001 | `mall` 是商品、订单、物流、售后、审批、退款的唯一业务事实来源；AI 不直连其数据库。 | AI 仅能经受控 API/工具读取或请求执行，避免模型或副本篡改业务事实。 |
| ADR-002 | Gateway、会话、AI 编排、知识、评测、洞察在单仓库中作为独立应用演进。 | 服务拥有独立数据库、构建、配置和部署单元；P1 只建骨架。 |
| ADR-003 | 跨服务异步事实使用 RabbitMQ + Outbox + Inbox；售后退款采用 `mall` 协调的 Saga。 | 生产和消费均须幂等，失败可重试、死信可审计；不引入 Seata。 |
| ADR-004 | 外部实时通信采用 REST、SSE、WebSocket 的组合。 | REST 提交和查询，SSE 推送 AI 流，WebSocket 处理人工协同与队列状态。 |
| ADR-005 | 知识服务使用 PostgreSQL + pgvector，实现关键词、向量和 RRF 混合检索。 | 文档、切片和向量归 `knowledge_db`；在线回答必须返回可追溯引用。 |
| ADR-006 | 售后操作采用低、中、高三级风险和确认/审批状态机。 | 模型只负责理解与收集信息；规则决定路径，用户确认和人工审批控制写操作。 |
| ADR-007 | 外部 JWT、内部服务 JWT 与可信租户上下文共同实现隔离。 | Gateway 清除伪造内部头；下游同时校验调用服务和用户身份快照。 |
| ADR-008 | Prompt、工作流、模型参数、规则和知识均按版本审核、灰度和回滚。 | 每个 AI 结果记录实际使用版本，旧业务申请固定使用创建时版本。 |
| ADR-009 | 前端采用 pnpm Monorepo，`chat-widget` 以组件包方式嵌入商城。 | 管理端初期经 Gateway 调用领域服务，不增加 BFF。 |
| ADR-010 | 本地开发使用 Docker Compose，生产目标为 Kubernetes。 | P1 提供本地基础设施；生产化阶段补充镜像、NetworkPolicy、扩缩容与灰度。 |
